# TRAE CN 高效工作规范

> 本项目永久生效的工作规范，所有会话必须严格遵守

---

## 一、核心命令与操作规范

### 1.1 内置快捷斜杠命令

请优先使用这些内置命令提升操作效率：

- `/clear` / `/reset`：当对话过长、上下文溢出时，主动清空对话历史并释放上下文，避免失忆
- `/model`：用于切换模型，严格遵守下方的模型调用规则
- `/init`：为当前项目初始化 AGENTS.md 配置文件，统一项目级工作规则
- `/status`：随时查看当前会话状态，确保工作环境正常
- `/feedback`：遇到无法解决的问题时，主动提交反馈

### 1.2 上下文精准引用规范

为了避免幻觉，请严格使用以下方式引用项目内容，禁止凭空猜测：

- 使用 `#` 符号精准引用：`#文件名` 引用文件、`#类名/函数名` 引用代码元素、`#文件夹` 引用目录
- 所有代码修改必须基于项目真实存在的文件和代码，禁止编造不存在的文件、函数或逻辑
- 主动帮我配置 `.traeignore` 文件，排除 `node_modules/`、`dist/`、`vendor/`、`.git/` 等无关目录，让你只关注核心代码，提升理解准确性

### 1.3 自定义命令支持

如果有重复的工作任务，你可以主动帮我创建自定义斜杠命令，将常用提示词固化为快捷操作，提升复用性。

## 二、执行能力与工作流程

### 2.1 任务拆分原则

- **禁止一次性处理过于复杂的大任务**，所有任务必须拆分为多个小步骤，一次只做一件事
- 每次任务开始前，先向我确认拆分后的步骤，先完成第一步，再进行下一步，禁止抢跑
- 例如：开发项目时，先做首页，再做列表页，再做功能页，逐个完成

### 2.2 明确的验收标准

- 所有任务必须有明确的完成定义，任务完成后，主动向我汇报每个验收项是否完成
- 禁止做到 "差不多" 就停止，必须逐项核对需求，确保所有要求都被满足
- 例如：开发功能后，主动说明 "已完成添加、删除、本地存储功能，移动端适配已完成"

### 2.3 沟通与修正规则

- 如果我对你的输出不满意，禁止重新开对话推翻重来，请在当前对话的基础上增量修改
- 修正时先明确指出问题，再给出调整方向，保留之前的上下文，提升效率
- 遇到不确定的需求，必须主动询问我确认，禁止自行猜测用户意图

---

## 三、代码质量保障规范

### 3.1 代码风格与规范

- 严格遵守项目已有的代码风格，禁止随意修改原有风格
- 变量、函数使用统一的命名规范（如驼峰命名），禁止混合风格
- 所有函数必须添加清晰的注释，关键逻辑必须添加行间注释，禁止无注释的 "面条代码"
- 文件编码统一使用 UTF-8，避免中文乱码问题

### 3.2 错误与安全处理

- 所有关键操作必须添加错误处理逻辑，禁止裸奔的代码
- 禁止使用原生 SQL，优先使用 ORM 框架，避免安全漏洞
- 代码修改前，必须先分析这个修改会影响哪些文件和函数，评估潜在的副作用
- 修改完成后，必须主动检查修改是否会影响其他功能，禁止 "改了 A 坏了 B"

### 3.3 版本与测试保障

- 每次完成一个功能，主动帮我提交 Git 代码，commit 信息清晰描述本次修改内容
- 重要修改前，提醒我提交代码备份，出问题可以快速回滚
- 生成的代码必须是可运行的，禁止生成有语法错误的代码，完成后主动进行基础的语法检查

## 四、模型调用与成本控制

### 4.1 模型优先级规则

- **优先使用 TRAE CN 内置的免费模型**：如 Doubao-Seed-2.0-Code、Qwen3、DeepSeek 等，这些模型对我来说完全免费，无调用限制
- 禁止默认使用我自定义配置的收费模型（如我自己添加的 OpenAI、Claude 等付费 API 模型）
- 只有当内置免费模型无法完成任务（如特别复杂的深度推理、超大型项目分析）时，你必须先向我申请："当前任务内置模型可能无法很好完成，是否允许使用自定义收费模型？"，得到我的明确同意后，才能切换到收费模型

### 4.2 资源节约规则

- 禁止无意义的重复调用模型，同一个任务不要反复重新生成
- 大任务拆分后，每次只处理一个小步骤，避免一次性消耗过多上下文资源
- 简单的问答、小代码修改，使用轻量的 Chat 模式即可，不需要启动重型的 Agent 或 Max 模式

## 五、幻觉与错误规避

### 5.1 禁止幻觉生成

- 严格基于我提供的上下文和项目真实代码工作，**禁止编造不存在的代码、函数、API 或数据**
- 如果你不确定某个信息，必须明确告诉我："这个信息我不确定，需要你提供更多细节"，禁止凭空生成错误内容
- 禁止过度生成代码，不要为了凑功能添加我没要求的额外功能，严格按需求来

### 5.2 避免常见问题

- **代码截断问题**：如果代码过长，禁止只输出一半就停止，主动使用 "继续" 机制，把完整的代码输出给我
- **上下文丢失问题**：如果对话过长，主动提醒我上下文即将溢出，建议拆分任务或者清空历史重新开始
- **答非所问问题**：如果发现你理解错了我的需求，主动停下来，重新确认需求，不要继续输出错误的内容
- **过度复杂问题**：禁止为简单任务过度设计，用最简单的方案解决问题，不要画蛇添足

### 5.3 错误自查机制

- 生成代码后，必须先自己检查一遍：有没有语法错误？有没有逻辑 bug？有没有安全问题？
- 如果发现错误，主动修正，不要把有问题的代码直接给我
- 如果遇到终端报错，主动分析错误栈，定位问题行，给出修复方案

## 六、其他注意事项

- 所有输出优先使用中文，保持清晰易懂
- 不要省略关键细节，不要用 "等等"、"类似的" 这种模糊的表述，把内容说清楚
- 如果遇到你无法解决的问题，直接告诉我，不要硬着头皮生成错误的内容
